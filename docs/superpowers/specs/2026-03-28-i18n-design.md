# i18n Design — Cyberpunk Combat Zone Companion

**Date:** 2026-03-28
**Status:** Approved
**Languages:** English (default), French

## Overview

Add bilingual support (EN/FR) to the app using a lightweight custom i18n system built on TypeScript + Zustand. No external i18n library. Supabase columns for catalog translations, typed translation files for UI strings.

### Translation Philosophy

- **Keep in English:** Proper nouns (faction names, character names, weapon names, item names, program names), mechanical keywords (Bulky, Accurate, Cybergear, Netrunner...), stat labels (EB, SC, HP)
- **Translate to French:** Descriptions, rules, glossary definitions, UI labels, buttons, section titles, validation messages, tooltips, toast notifications
- **Case-by-case gameplay terms:** Some terms stay English if idiomatic (Gonk, Leader), others get French equivalents (Characters → Personnages, Specialists → Spécialistes)

## 1. Infrastructure i18n

### Locale in Zustand Store

New field in `StoreState`:

```ts
locale: 'en' | 'fr'  // default: 'en', persisted in localStorage
```

### Translation Files

```
src/i18n/
  types.ts        → type Translations (all keys, flat namespaced)
  en.ts           → export const en: Translations = { ... }
  fr.ts           → export const fr: Translations = { ... }
  index.ts        → useT() hook + useLocalized() hook
```

### Key Structure — flat namespaces

```ts
type Translations = {
  // nav
  'nav.hq': string;
  'nav.database': string;
  'nav.play': string;
  // validation
  'validation.missingLeader': string;
  'validation.budgetExceeded': string;
  // settings
  'settings.language': string;
  'settings.cardsPerRow': string;
  // types
  'types.leader': string;
  'types.character': string;
  'types.gonk': string;
  'types.specialist': string;
  'types.drone': string;
  // ... ~150-200 keys total
}
```

### `useT()` Hook

```ts
function useT(): (key: keyof Translations, vars?: Record<string, string | number>) => string
```

- Reads `locale` from Zustand store
- Simple interpolation: `t('validation.budgetExceeded', { cost: 150, budget: 100 })`
  - Pattern: `"Budget exceeded! ({cost}/{budget} EB)"`
- Fallback to `en` if key missing in `fr`

### `useLocalized()` Hook

```ts
function useLocalized(): <T>(item: T, field: string) => string
```

- Reads `locale` from store
- If `locale === 'fr'` and `item[field + '_fr']` exists and is non-empty → return it
- Otherwise → return `item[field]`
- Used for catalog data: `loc(weapon, 'description')`, `loc(entry, 'definition')`

### Why Two Separate Tools

- `t()` = compile-time known keys, exhaustively typed, for UI strings
- `localized()` = dynamic resolution on data objects with progressive translations
- Clean separation, no mixing

## 2. Catalog Data (Supabase + seed.ts)

### New Supabase Columns

| Table | New columns |
|---|---|
| `factions` | `description_fr` |
| `profiles` | `description_fr` |
| `weapons` | `description_fr` |
| `items` | `description_fr` |
| `programs` | `description_fr` |
| `objectives` | `name_fr`, `description_fr` |
| `loots` | `name_fr`, `description_fr` |

All new columns: `TEXT`, nullable, default `NULL`.

**Not translated:**
- `lineages` — `name` is a proper noun, no description field
- `keywords` — mechanical rulebook terms, stay English
- `name` on most tables — proper Cyberpunk nouns

**Exception:** `objectives` and `loots` have generic names that warrant translation → `name_fr`.

### Impact on TypeScript Types

Each catalog type gains an optional field:

```ts
interface Weapon {
  // ... existing fields
  description_fr?: string;
}
```

Optional (`?`) so untranslated entries fall back to English gracefully.

### Impact on `useCatalog.ts`

Mapper functions (`mapWeapon`, `mapFaction`, etc.) map the new columns. No locale resolution in mappers — resolution happens at the component level via `useLocalized()`.

### Impact on `seed.ts`

Each entry gains `description_fr: ""` (empty string, filled progressively). This ensures the fallback seed data matches the Supabase schema.

### Glossary (`glossary.ts`)

Each `GlossaryEntry` gains:

```ts
interface GlossaryEntry {
  term: string;           // stays English (rulebook term)
  definition: string;     // English definition
  definition_fr?: string; // French definition
  page?: string;
  highlight?: boolean;
}
```

## 3. UI Language Selector

Located in **DisplaySettings** (existing gear icon panel), at the top:

- Simple toggle: `EN` / `FR`
- Persisted in Zustand store → localStorage
- Changing language instantly re-renders all components (Zustand reactivity)

## 4. Component Translation Scope

### Priority 1 — Navigation & Layout
- Nav bar tab titles
- DisplaySettings labels
- SectionHeader titles

### Priority 2 — HQ & Campaign
- CampaignManager, RosterList, TeamBuilder
- Validation messages (`validation.ts` → all ~10 messages)
- PostGameDialog (4 wizard steps)
- Toast notifications

### Priority 3 — Database
- FactionsTab, ModelsTab, ItemsTab, ArmoryContent, ActionsContent
- ObjectivesContent, LootsContent, ProgramCard
- Filter labels, admin buttons
- TYPE_LABEL maps

### Priority 4 — Play
- ActiveMatchView, CharacterCard labels
- MercCard, WeaponTile
- GlitchCanvas — no text, skipped

### Not Translated
- Proper nouns (factions, characters, weapons, items, programs)
- Mechanical keywords (Bulky, Accurate, Cybergear, Netrunner...)
- Numeric/stat labels (EB, SC, HP)
- Image content

### Estimated Volume
- ~150-200 UI translation keys
- ~200+ catalog descriptions to translate in Supabase

## 5. Migration Strategy

Translations are filled **progressively** — the app works fully in English from day 1 of the implementation. French translations are added incrementally:

1. First: infrastructure (store, hooks, translation files with EN keys)
2. Then: UI strings (component by component, priority order)
3. Then: Supabase migration (add columns)
4. Then: catalog translations (can be done over time, fallback to EN)

No big-bang required. The app never breaks — untranslated content shows English.
